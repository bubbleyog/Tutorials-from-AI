# 第三章示例代码详解

Slurm 是现代 HPC 集群的标准配置，其设计逻辑非常严谨。本章通过示例带你理解 Slurm 的核心概念。

## 示例 1: 基础作业 (`01_basic_job.slurm`)

最简单的串行作业。

### Slurm vs PBS: 路径处理
*   **Slurm**: 自动保留提交时的目录。如果你在 `~/projects/test` 下执行 `sbatch`，作业开始运行时的目录就是 `~/projects/test`。
*   **PBS**: 默认在 `~` 目录。需要 `cd $PBS_O_WORKDIR`。
*   因此，Slurm 脚本通常比 PBS 脚本少一行 `cd` 命令，显得更简洁。

## 示例 2: MPI 任务 (`02_mpi_job.slurm`)

### 核心参数：`ntasks`
*   `--ntasks=40`: 这里的 Task，在 MPI 语境下就是 Rank（进程）。
*   `--nodes=2`: 告诉调度器，这 40 个进程必须分散在 2 个物理节点上（可能是为了更大的内存带宽，或者单节点核不够）。
*   **`srun` 的魔法**: 当你在脚本里直接运行 `srun ./prog` 时，`srun` 会读取环境变量，知道你要跑 40 个进程，于是自动帮你拉起 MPI 运行时环境。这比手动写 `mpirun -np 40 -hostfile ...` 要方便且健壮得多。

## 示例 3: OpenMP / 多线程任务 (`03_openmp_job.slurm`)

### 核心参数：`cpus-per-task`
*   `--ntasks=1`: 我们只运行 **1** 个进程（主程序）。
*   `--cpus-per-task=20`: 但是这个进程很特别，它需要 20 个 CPU 核心来跑多线程。
*   **常见错误**:
    *   如果写成 `--ntasks=20`，Slurm 会以为你要跑 20 个独立的进程。这对于 MPI 没错，但对于 OpenMP，可能会导致资源分配逻辑混乱（例如分配到了不同节点上，但 OpenMP 无法跨节点）。
*   **环境变量**: `export OMP_NUM_THREADS=$SLURM_CPUS_PER_TASK` 是一种不需要修改数字的通用写法。

## 示例 4: GPU 任务 (`04_gpu_job.slurm`)

### 通用资源 (GRES)
*   `--gres=gpu:1`: 这是 Slurm 申请 GPU 的标准语法。
*   有时你会看到 `--gres=gpu:v100:1`，这意味着明确指定申请 V100 型号的卡。
*   **Topology Awareness (拓扑感知)**: Slurm 非常聪明，如果你申请了 GPU，它会自动把你分配到离 GPU 最近的 CPU 核心上，以保证数据传输速度。

## 示例 5: 数组作业 (`05_array_job.slurm`)

这是处理“大规模并发同质任务”的神器。

*   **场景**: 你有 10,000 张图片需要处理，处理逻辑一样，只是文件名不同。
*   **笨办法**: 写一个循环提交 10,000 次 `sbatch`。这会让调度对列爆炸，管理员会封你的号。
*   **好办法**: 提交 **1** 个数组作业，包含 10,000 个子任务。
*   `--array=1-10000`: 指定索引范围。
*   `%A_%a`: 在输出文件名中使用。`%A` 是主 ID（如 1001），`%a` 是子索引（如 1, 2, 3...）。输出文件会变成 `array_1001_1.out`, `array_1001_2.out`... 井井有条。
