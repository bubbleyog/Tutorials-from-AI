# 第二章示例代码详解

本章提供的示例代码位于 `examples/` 目录下。PBS 作业脚本是连接用户与超级计算机资源的桥梁，理解其中的每一行指令至关重要。

## 示例 1: 基础作业 (`01_basic_job.pbs`)

这是最常用的单核串行作业模板，适用于数据预处理、简单的 Python 脚本等。

### 关键指令解析

*   **`#PBS -l nodes=1:ppn=1`**:
    *   `nodes=1`: 只要 1 台机器。
    *   `ppn=1`: Processors Per Node，即每台机器上只用 1 个核心。
    *   如果你是单线程程序，千万不要申请 `ppn=20`，这会浪费那 19 个核心，导致通过率降低（因为系统很难凑出这么大的空闲块给你）。

*   **`cd $PBS_O_WORKDIR`**:
    *   **最容易被遗忘的命令！**
    *   当你提交作业时（例如在 `~/project/run1` 目录下执行 `qsub`），作业实际启动时的默认目录通常是你的主目录（`~`）。
    *   如果不加这行，你的程序可能找不到当前目录下的数据文件，直接报错 "File not found"。
    *   `$PBS_O_WORKDIR` 是 PBS 系统定义的环境变量，存储了你执行 `qsub` 命令时的路径。

## 示例 2: MPI 并行作业 (`02_mpi_job.pbs`)

用于运行跨多个物理节点的并行程序（如 VASP, LAMMPS, GROMACS）。

### 核心机制

*   **资源申请**: `nodes=2:ppn=20` 表示你需要 2 台机器，每台机器都要独占 20 个核。总共获得了 40 个核。
*   **`$PBS_NODEFILE`**:
    *   这是一个文本文件的路径。
    *   系统在分配机器后，会把机器名写入这个文件（例如里面写着 20 行 node01，20 行 node02）。
    *   `mpirun` 会读取这个文件，知道该把任务发给谁。

## 示例 3: OpenMP 多线程作业 (`03_openmp_job.pbs`)

用于运行单节点内多线程并行的程序（如 Gaussian, MATLAB, 以及许多 Python 科学计算库）。

### 关键点

*   **SMP (Symmetric Multi-Processing)**: OpenMP 程序只能在**同一台**机器的内存中跑，不能跨节点。所以通常写 `nodes=1:ppn=N`。
*   **`export OMP_NUM_THREADS=$PBS_NUM_PPN`**:
    *   如果不限制线程数，程序可能会生成成百上千个线程，导致 CPU 频繁切换上下文（Context Switch），性能反而下降。
    *   将其设置为申请到的核心数是最佳实践。

## 示例 4: GPU 作业 (`04_gpu_job.pbs`)

适用于深度学习（TensorFlow, PyTorch）或 GPU 加速的科学计算软件。

### 资源语法

*   `nodes=1:ppn=1:gpus=1`: 这是一个典型的语法，但不同计算中心的 PBS 配置可能略有不同。有的可能是 `-l gpus=1`。请参考具体集群的文档。
*   **`nvidia-smi`**: 这是一个很有用的调试命令。在作业开始前运行它，可以在日志里确认：
    1.  你真的分到了 GPU 吗？
    2.  分到的是哪种型号（A100, V100）？
    3.  Driver 版本是否满足你的 CUDA 要求？

## 示例 5: 交互式作业 (`05_interactive.sh`)

这不是一个提交脚本，而是一个命令行指令。

### 使用场景
*   **错误做法**: 在登录节点直接运行 `python train.py`。这会卡顿所有人的操作。
*   **正确做法**: `qsub -I ...` 进入计算节点，然后运行 `python train.py`。

### 注意事项
*   交互式作业也会占用资源并计费。
*   如果你退出了终端，作业就结束了，所有临时进程会被杀掉。如果需要长时间跑任务，还是请用 `qsub` 提交脚本。
